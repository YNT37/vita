# Vita · AI 生活管家 —— 功能需求文档（FRD v1.1）

| 项 | 内容 |
|---|---|
| 项目名称 | Vita · AI 生活管家 |
| 版本 | v1.1（含多用户账号体系） |
| 技术栈 | Next.js(App Router+TS+Tailwind) / Flask + SQLAlchemy / PostgreSQL / LangChain + DeepSeek / JWT |
| 文档目的 | 定义系统功能与验收标准，指导开发与考核 |

## 1. 项目背景与目标
居家实训考核作品。做一个**多用户、具备情绪价值的个人生活管理工具**：用户注册登录后，
可选择不同 AI 角色（管家/奴才/毒舌闺蜜/暖心恋人），由其以对应语气协助**记账、日程提醒**，
并提供**每日播报、自然语言快捷录入、对话陪伴**。产品上线可访问，数据按用户隔离，工程规范可追溯。

## 2. 用户与使用场景
- **目标用户**：需要轻量管理收支与琐事、并希望有陪伴感的个人用户（多账号，各自独立数据）。
- **典型场景**：注册登录 → 仪表盘看 AI 今日播报 → "午饭30"一句话记账 → 睡前看收支图表并与角色对话 → 设置账单提醒。

## 3. 功能范围
| 范围 | 内容 |
|---|---|
| ✅ 本期实现(MVP) | 账号与鉴权、记账理财、日程提醒、AI 管家（角色/对话/简报/自然语言录入）、仪表盘 |
| 🔲 暂不实现 | 习惯打卡、心情日记、消息推送、第三方登录(OAuth)、移动端原生 App |

## 4. 功能需求详述

### 4.1 账号与鉴权（横切基础）
方案：JWT 无状态鉴权；密码用 `werkzeug.security` 哈希存储。

| 编号 | 功能 | 说明 | 规则/异常 |
|---|---|---|---|
| FR-0.1 | 注册 | 用户名 + 密码注册 | 用户名唯一；密码 ≥6 位；重复→409；非法→400 |
| FR-0.2 | 登录 | 校验后签发 JWT | 密码错误→401 |
| FR-0.3 | 登出 | 前端清除 token | — |
| FR-0.4 | 当前用户 | `GET /api/me` 返回当前用户 | 无/失效 token→401 |
| FR-0.5 | 接口鉴权 | 所有业务接口需有效 JWT | 未授权→401 |
| FR-0.6 | 数据隔离 | 用户只能访问/操作自己的数据 | 越权→404 |

### 4.2 记账理财
| 编号 | 功能 | 说明 | 规则/异常 |
|---|---|---|---|
| FR-1.1 | 新增交易 | 金额、类型、分类、备注、日期 | 金额数字且>0；type∈{income,expense}；日期合法；非法→400 |
| FR-1.2 | 交易列表 | 按月倒序，支持分类筛选 | 仅当前用户数据 |
| FR-1.3 | 删除交易 | 删除指定记录 | 不存在/越权→404 |
| FR-1.4 | 收支统计 | 按月汇总收入/支出/分类占比/每日趋势 | 仅当前用户 |

### 4.3 日程提醒
| 编号 | 功能 | 说明 | 规则/异常 |
|---|---|---|---|
| FR-2.1 | 新增提醒 | 标题、到期日、类型、备注 | 标题非空；日期合法 |
| FR-2.2 | 提醒列表 | 按到期日升序，临期高亮 | 仅当前用户 |
| FR-2.3 | 完成切换 | 勾选切换 done | 不存在/越权→404 |
| FR-2.4 | 删除提醒 | 删除指定提醒 | 不存在/越权→404 |

### 4.4 AI 管家（核心亮点）
| 编号 | 功能 | 说明 | 规则/异常 |
|---|---|---|---|
| FR-3.1 | 角色切换 | 管家/奴才/毒舌闺蜜/暖心恋人，语气随之变化 | 未知角色→拒绝；按用户保存偏好 |
| FR-3.2 | 角色对话 | 以当前角色语气对话并存历史 | 空消息→提示 |
| FR-3.3 | 每日简报 | 聚合当日待办+账单+消费生成播报 | 无数据→通用问候 |
| FR-3.4 | 自然语言录入 | 对话中识别记账/提醒/余额 → 弹出可编辑确认卡 → 用户点确定后写入 | 解析失败不弹卡；取消不写入 |
| FR-3.5 | AI 降级 | 无 Key/超时/异常/非法返回→角色静态兜底文案，绝不崩 | 系统健壮性要求 |

### 4.5 仪表盘
| 编号 | 功能 |
|---|---|
| FR-4.1 | 本月收支概览卡片 |
| FR-4.2 | 收支/分类/趋势图表 |
| FR-4.3 | AI 每日播报区（FR-3.3） |
| FR-4.4 | 今日到期提醒 |

## 5. 非功能需求
| 类别 | 要求 |
|---|---|
| 安全 | 密码仅存哈希；`JWT_SECRET`/密钥走环境变量；token 设过期；所有业务接口校验 JWT；数据库不对公网暴露 |
| 健壮性 | 输入校验；空/超长/非法/重复提交不崩；异常统一返回规范错误体，绝不裸崩 500 |
| 性能 | 常规请求 <1s；AI 请求设超时并降级 |
| 可用性 | 提供可点击线上 URL；UI 响应式、简洁美观 |
| 可维护性 | 前后端分离、ORM、错误集中处理、目录清晰 |

## 6. 数据模型（实体）
| 实体 | 关键字段 |
|---|---|
| users | id, username(unique), password_hash, created_at |
| categories | id, name, kind(expense/income) |
| transactions | id, user_id, type, amount, category, note, date, created_at |
| reminders | id, user_id, title, due_at, type, done, note, created_at |
| chat_messages | id, user_id, role, content, persona, created_at |
| settings | id, user_id, key, value |

## 7. 接口需求（概述）
- 鉴权：`POST /api/auth/register`、`POST /api/auth/login`、`GET /api/me`
- 记账：`POST/GET/DELETE /api/transactions`、`GET /api/stats/summary`
- 提醒：`POST/GET/PATCH/DELETE /api/reminders`
- AI：`GET/POST /api/persona`、`POST /api/ai/chat`、`POST /api/ai/brief`、`POST /api/ai/parse`
- 健康：`GET /api/health`
- 除 register/login/health 外，所有接口需 `Authorization: Bearer <token>`，并按 user_id 隔离数据。

| 路由 | 页面 | 访问 |
|---|---|---|
| `/login` | 登录 | 公开 |
| `/register` | 注册 | 公开 |
| `/` | AI 管家主页 | 需登录 |
| `/stats` | 统计中心 | 需登录 |
| `/records` | 记账 | 需登录 |
| `/reminders` | 提醒 | 需登录 |
| `/user` | 我的（账号中心） | 需登录 |

底栏五栏：管家 · 统计 · 记账 · 提醒 · 我的。未登录访问受保护页 → 跳转 `/login`。

## 9. 验收标准（对应考核）
- [ ] 前端 ≥3 路由（实际 6）、后端 ≥3 API、含数据库
- [ ] 多用户可注册登录，数据按用户隔离，接口鉴权生效
- [ ] 线上 URL 可访问，接口正常响应
- [ ] 每功能点独立提交、Commit 规范、跨 ≥3 天
- [ ] 健壮性场景覆盖（空/非法/超长/越权/未授权）
- [ ] 交付物齐全：README、API 文档、prompt_log、研发日志、个人总结、演示录屏、AI Code Review 截图
- [ ] 加分：单元测试、环境变量、Docker/CI/CD

## 10. 约束与假设
- 多用户，JWT 鉴权，无第三方登录（MVP）。
- AI 依赖 DeepSeek，异常时降级文案保证可用。
- 部署：后端自建云服务器(Docker)，前端 Vercel，数据库 PostgreSQL。
